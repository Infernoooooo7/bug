PHISHING DETECTION CORE LOGIC
=============================

This file explains the current detection flow implemented in the API.

0) API CORS MIDDLEWARE
----------------------
The API enables CORS with:

- allow_origins = ["*"]
- allow_credentials = True
- allow_methods = ["*"]
- allow_headers = ["*"]

This allows frontend apps to call the backend from browser contexts.

1) INPUT MODEL
--------------
The API expects a JSON body with one field:

- content: full email/message text to analyze

2) URL EXTRACTION
-----------------
The detector extracts all HTTP/HTTPS links using:

- https?://[^\s]+

3) URL EXPANSION (REDIRECT RESOLUTION)
--------------------------------------
Each URL is requested with:

- timeout = 5 seconds
- allow_redirects = True

If successful, the final redirect destination is used.
If failed (timeout/network error), the original URL is used.

4) DOMAIN EXTRACTION HELPERS
----------------------------
Two helpers are used:

- get_base_domain(url): returns domain.suffix (example.com)
- extract_domain(email): returns email domain (example.com)

5) URL SIGNALS DETECTED
-----------------------
For each expanded URL, the API computes:

- unicode_homograph:
  True if any URL character has ASCII code > 127
- visual_homograph:
  True when base domain is highly similar to sender domain
  but not exactly equal
- shortened_url:
  True if original URL contains known shortener hosts
- external_link:
  True if sender domain is not part of destination base domain
- suspicious_domain:
  True if expanded URL text contains risky keywords

6) VISUAL HOMOGRAPH CHECK
-------------------------
If sender domain exists:

- similarity_score = SequenceMatcher(base_domain, sender_domain).ratio()
- visual_homograph = True when:
  - base_domain != sender_domain
  - similarity_score > 0.8

The API returns similarity_score rounded to 2 decimals.

7) URL RISK SCORING (WEIGHTED)
------------------------------
Each active signal adds weighted points:

- +50 for visual_homograph
- +40 for unicode_homograph
- +15 for shortened_url
- if current score > 0: +5 for external_link
- if current score > 0: +5 for suspicious_domain

Final URL risk is capped at 100.

This means external_link and suspicious_domain do not raise a zero score by themselves.

8) PER-URL EXPLANATION GENERATION
---------------------------------
Explanation reasons are appended in this order:

- domain closely mimics sender domain
- uses Unicode characters
- shortened URL hides destination
- points to external domain (only when risk > 0)
- contains phishing-related keywords (only when risk > 0)

If no signals are active:

- No major phishing indicators detected.

9) URL CLASSIFICATION
---------------------
URLs are grouped as:

- risky_urls if risk_percent > 0
- safe_urls if risk_percent == 0

10) EMAIL HEADER EXTRACTION
---------------------------
The content is scanned for these lines:

- From:
- Return-Path:

If present, values are extracted as sender and return_path.

11) SENDER SPOOF CHECK
----------------------
Spoofing is checked by comparing extracted domains:

- sender_domain = extract_domain(sender)
- return_domain = extract_domain(return_path)
- spoofed = (sender_domain != return_domain)

If either domain is missing, spoofed is False.

12) EMAIL RISK CALCULATION
--------------------------
Email risk uses URL average plus spoof penalty:

1. total_risk = sum of per-URL risk values
2. avg_risk = int(total_risk / len(urls)) if URLs exist, else 0
3. email_risk = avg_risk
4. if spoofed is True, add 25 points
5. cap email_risk at 100

13) RISK LEVEL MAPPING
----------------------
email_risk_percent is mapped to:

- 0 to 29 -> risk_level = low, risk_color = green
- 30 to 69 -> risk_level = medium, risk_color = yellow
- 70 to 100 -> risk_level = high, risk_color = red

14) RESPONSE SHAPE
------------------
The API returns:

- total_urls
- risky_count
- safe_count
- email_risk_percent
- risk_level
- risk_color
- email_analysis:
  - sender
  - return_path
  - spoofed
- risky_urls
- safe_urls

Each URL object contains:

- original_url
- expanded_url
- risk_percent
- visual_homograph
- similarity_score
- unicode_homograph
- shortened_url
- external_link
- suspicious_domain
- explanation

15) CURRENT LIMITATIONS
-----------------------
- Rule-based heuristics only (no ML model)
- Static keyword and shortener lists
- Similarity threshold is rule-based and may produce false positives/negatives
- External and keyword signals only contribute after risk is already above zero
- Header extraction uses simple regex parsing
- URL expansion depends on network availability/timeouts
- No SPF/DKIM/DMARC validation
- No threat-intelligence reputation integration
